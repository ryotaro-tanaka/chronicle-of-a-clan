# Stage 3 — coc init (slot-based saves) + CI + Core/UI boundary cleanup

## Goal
Improve first-run experience and developer workflow by adding:
- `coc init <save_dir>` to create a new save slot from a template
- clearer no-arg usage messages (including `init`)
- CI: run `go test ./...` on pull requests
- a small refactor to restore Core/UI responsibility boundaries (move status formatting out of core/save)

## Non-goals
- No quest generation or quest listing
- No hiring/dismissal
- No equipment crafting/upgrade flows
- No time advancement
- No balancing work
- No help command
- No TUI screens

## Decisions

### Saves directory (slot-based)
- All save slots live under the project-root `saves/` directory.
- `<save_dir>` is a slot name (not a path).
- The program must not accept `<save_dir>` as an arbitrary filesystem path.

### Slot name validation
- Allowed characters: `A-Za-z0-9._-`
- Must not start with `-`
- Any other characters are rejected with an actionable error.

### Template directory
- The init template is committed at:
  - `examples/save_init_template/`
- At minimum, it contains:
  - `examples/save_init_template/clan.json` (valid, supported `save_version`)

### Build artifacts
- Build outputs are placed under `bin/` (and `bin/` is ignored by git).

### Core/UI boundary (Stage 3 refactor)
- `internal/core/` must not format user-facing text output.
- UI string formatting must live under `internal/ui/` (e.g., `internal/ui/format/`).
- Core may expose a small read-only view model for status (data only).

## Definition of Done (DoD)

### coc init
- `coc init <save_dir>` creates `saves/<save_dir>/` and copies the template contents into it.
- If the slot directory already exists: fail with an actionable error (no overwrite in Stage 3).
- If `<save_dir>` is invalid per validation rules: fail with an actionable error.
- After `init`, running `coc <save_dir>` works without additional manual steps.

### No-arg messages
- `coc` (no args) prints:
  - `Usage: coc <save_dir>`
  - `Usage: coc init <save_dir>`

Optional (nice-to-have, not required):
- `coc init` (no args) prints usage and lists existing slots under `saves/` if easy.

### CI (GitHub Actions)
- Add a workflow that runs on pull requests:
  - `go test ./...`

### Core/UI boundary cleanup
- Remove UI-facing string formatting from `internal/core/save/`.
- Implement a data-only status view model in Core (example shape):
  - `core/status.StatusView{ClanName, Day, Gold, Fame, MembersCount, ActiveQuestsCount}`
- Implement status string formatting in UI:
  - UI renders the 3-line contract from the view model:
    - `Clan: <name>   Day: <day>`
    - `Gold: <gold>   Fame: <fame>`
    - `Members: <count>   ActiveQuests: <count>`
- Update tests accordingly:
  - Core tests validate the view model extraction.
  - UI tests validate formatting output from a given view model.

## Test Checklist
- `coc init <name>` creates `saves/<name>/` with expected files.
- `coc init <name>` fails when `saves/<name>/` already exists.
- `coc init <name>` fails for invalid names (bad chars, starts with `-`).
- CI workflow runs `go test ./...` on PR.
- Status output still matches the 3-line contract.
